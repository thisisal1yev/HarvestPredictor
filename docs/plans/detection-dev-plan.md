## 1. Revised Feature Spec (Post-Debate Consensus)

### Naming
- **Rename "Detection" to "Diagnosis"** throughout the UI and codebase
- Route: `/dashboard/diagnosis` (not `/dashboard/cv/detection`)
- Reason: "Diagnosis" communicates value to farmers. "Detection" is ML jargon.

### Page Structure — 3 Tabs

| Tab | Route | Content | Visible To |
|-----|-------|---------|------------|
| **Diagnose** | `/dashboard/diagnosis` (default) | Photo upload + results + treatment | All users |
| **History** | `/dashboard/diagnosis?tab=history` | Session timeline with filters | All users |
| **Models** | `/dashboard/diagnosis?tab=models` | Model CRUD (upload, delete, set default) | Admin only |

### What Changed from Original Spec

| Original Spec | Revised (Post-Debate) | Reason |
|---------------|----------------------|--------|
| Single page with 4 zones | 3 tabs (Diagnose/History/Models) | UX analysis: information overload |
| Photo + Video + IP Stream | **Photo only** for MVP | Value analysis: video/stream = different product, near-zero farmer value |
| Manual model selection dropdown | **Auto-select by crop type**, override available | UX + Value: farmers don't know model names |
| Raw bounding box coordinates | Visual canvas overlay + detection list | Competitor analysis: industry standard pattern |
| No treatment recommendations | **Treatment inline with each detection** | Value analysis: #1 missing feature |
| All confidence levels shown | **Severity tiers**: Confirmed/Likely/Possible | Value analysis: <50% results confuse farmers |
| `.pt` and `.onnx` model support | **ONNX-only** model policy | Security analysis: `.pt` = RCE via pickle (CRITICAL) |
| No auto-detect | **Auto-detect on upload** (zero-click) | Competitor analysis: Ultralytics HUB pattern |
| Store annotated crops 72h | **Thumbnails in history**, delete originals immediately | Security + storage optimization |
| JPEG base64 over WebSocket | **Binary WS frames** (for future streaming) | Tech analysis: 33% bandwidth savings |
| No empty states | Full empty state design for every scenario | UX analysis: critical gap |
| No error handling defined | Error states for every failure mode | UX analysis: critical gap |

### MVP Feature Set

**Include:**
1. Single photo upload (drag-and-drop + file picker + mobile camera)
2. Auto-detect on upload (zero-click inference)
3. Auto-model-select by crop type (from user's farm/field context)
4. Detection results: canvas overlay with bounding boxes + detection list
5. Severity tiers: Confirmed (90%+), Likely (70-89%), Possible (50-69%), hidden (<50%)
6. Treatment recommendations inline with each detection (from KnowledgeBase)
7. Detection history as session-based timeline
8. Thumbnails in history (generated during processing, originals deleted)
9. Empty states for: no models, no history, no detections found
10. Error states for: service down, upload failed, unsupported format
11. Model management (admin-only tab): upload .onnx, delete, set default per crop
12. i18n: disease names and recommendations in Uzbek

**Cut from MVP (Phase 2+):**
- Video file processing
- IP/RTSP/RTMP streaming
- Batch upload (>1 photo)
- SAHI tiling for drone imagery
- Offline photo queue
- Multi-model comparison

## 2. Architecture

### System Flow

```
┌─────────────┐      ┌──────────────────┐      ┌─────────────────┐
│  Nuxt 4     │      │  Nitro API       │      │  Python FastAPI  │
│  Frontend   │─────>│  Routes          │─────>│  CV Service      │
│  (Vue 3)    │      │  (server/api/cv/) │      │  (port 8100)     │
└─────────────┘      └──────────────────┘      └─────────────────┘
                            │                          │
                            v                          v
                     ┌──────────────┐          ┌──────────────┐
                     │  PostgreSQL  │          │  ONNX Runtime │
                     │  (Prisma)    │          │  + YOLO model │
                     └──────────────┘          └──────────────┘
```

### File Flow

- **Photos (<20MB):** Browser → Nitro API (proxy via `proxyRequest()`) → FastAPI → process → return results
- **Future large files (>20MB):** Browser → FastAPI directly (presigned token from Nitro)
- **Thumbnails:** Generated by FastAPI during processing, returned as base64 in response, stored in DB as `bytea` or on disk

### Key Architectural Decisions

1. **ONNX-only model format** — eliminates CRITICAL RCE risk from `.pt` pickle deserialization
2. **Photos proxied through Nitro** — maintains auth boundary, acceptable memory cost for <20MB files
3. **FastAPI sync endpoints** — Starlette threadpool handles concurrency (no `async def` for inference)
4. **LRU model cache (max 3)** — preload default model on startup, evict least-recently-used
5. **Auth at Nuxt layer** — CV service accepts only requests with valid `X-API-Key` header from Nitro
6. **Resize to 1280x1280** — default preprocessing for all photos (SAHI tiling deferred to Phase 2)

### CV Service Internal Architecture

```
cv-service/
  app/
    main.py                 # FastAPI app, lifespan, middleware
    config.py               # Settings (API key, model dir, limits)
    routers/
      detect.py             # POST /detect/image — single image detection
      models.py             # GET/POST/DELETE /models — model CRUD
      health.py             # GET /health — service health check
    services/
      detector.py           # YOLO inference wrapper
      model_manager.py      # LRU cache, model loading, validation
      image_processor.py    # Resize, EXIF strip, thumbnail generation
      onnx_validator.py     # ONNX file validation pipeline
    models/
      schemas.py            # Pydantic request/response schemas
  requirements.txt
  Dockerfile
```

## 3. Prisma Schema Changes

### New Models to Add

```prisma
model CVModel {
  id          String   @id @default(cuid())
  name        String
  filename    String   // UUID-generated filename on disk
  originalName String  // Original upload name (display only)
  format      String   @default("onnx") // onnx only
  cropType    String?  // cotton, wheat, tomato, etc. — for auto-select
  isDefault   Boolean  @default(false)
  fileSize    Int      // bytes
  hash        String   // SHA-256 for integrity
  metadata    Json?    // opset version, param count, input shape
  uploadedBy  String   // userId of admin who uploaded
  createdAt   DateTime @default(now())
  updatedAt   DateTime @updatedAt
  sessions    DetectionSession[]
}

model DetectionSession {
  id          String   @id @default(cuid())
  sourceType  String   @default("photo") // photo (video/stream in Phase 2)
  status      String   @default("processing") // processing, completed, failed
  imageCount  Int      @default(1)
  thumbnail   Bytes?   // JPEG thumbnail for history display
  processingMs Int?    // total processing time
  modelId     String
  userId      String
  fieldId     String?  // optional link to user's field
  createdAt   DateTime @default(now())
  updatedAt   DateTime @updatedAt
  model       CVModel  @relation(fields: [modelId], references: [id])
  detections  Detection[]

  @@index([userId, createdAt(sort: Desc)])
  @@index([modelId])
}

model Detection {
  id          String   @id @default(cuid())
  className   String   // disease/pest/weed name
  category    String   // disease, pest, weed, healthy
  confidence  Float    // 0.0-1.0
  severity    String   // confirmed, likely, possible
  bbox        Json     // {x: float, y: float, w: float, h: float} normalized 0-1
  sessionId   String
  createdAt   DateTime @default(now())
  session     DetectionSession @relation(fields: [sessionId], references: [id], onDelete: Cascade)

  @@index([sessionId])
  @@index([className])
  @@index([createdAt(sort: Desc)])
}

model KnowledgeBase {
  id            String   @id @default(cuid())
  diseaseName   String   @unique // English canonical name
  diseaseNameUz String?  // Uzbek name
  category      String   // disease, pest, weed
  symptoms      String   // description of visible symptoms
  symptomsUz    String?
  treatment     String   // recommended treatment steps
  treatmentUz   String?
  prevention    String?  // preventive measures
  preventionUz  String?
  severity      String   @default("medium") // low, medium, high, critical
  cropTypes     String[] // which crops this affects
  createdAt     DateTime @default(now())
  updatedAt     DateTime @updatedAt
}
```

### Changes to Existing Models

None. The CV models are standalone — they reference `userId` as a string (not a Prisma relation) to keep the CV module decoupled. This allows the CV service to be extracted as a fully independent microservice later.

## 7. Security Checklist

### Before Phase 1 Ships (CV Service)

- [ ] API key middleware on ALL endpoints (except `/health`)
- [ ] ONNX-only model policy enforced (reject `.pt` files at upload)
- [ ] ONNX validation pipeline: magic bytes, `onnx.checker.check_model()`, tensor dimension limits, param count limit
- [ ] UUID filenames for uploaded models (never use original filename on disk)
- [ ] SHA-256 hash computed and stored for each model
- [ ] File size limit enforced: 500MB max for models
- [ ] Temp file cleanup: `try/finally` + startup cleanup + `/dev/shm` for processing
- [ ] EXIF metadata stripped from uploaded images
- [ ] Image re-encoding before processing (prevents polyglot attacks)
- [ ] Magic byte validation for images (JPEG: `FF D8 FF`, PNG: `89 50 4E 47`)
- [ ] Model inference runs with memory limit (prevent OOM bombs)
- [ ] Test inference on upload: verify model can process a dummy 1x3x640x640 input

### Before Phase 2 Ships (Nuxt API)

- [ ] `requireUserSession(event)` on EVERY CV API route
- [ ] `requireAdmin(user)` on model upload/delete/update routes
- [ ] `userId` filter on all session/detection queries (ownership check)
- [ ] Request body size limit: 20MB for images at Nitro level
- [ ] Rate limiting: 30 detections/minute per user
- [ ] `X-API-Key` header added to all CV service proxy requests
- [ ] CV service URL not exposed to frontend (only Nitro knows it)

### Before Phase 3-4 Ships (Frontend)

- [ ] File type validation client-side (accept="image/jpeg,image/png")
- [ ] File size validation client-side (<20MB)
- [ ] No raw EXIF/metadata rendered in UI (XSS prevention)
- [ ] Detection results sanitized before rendering (className, category from allowlist)
- [ ] No CV service URL visible in client-side JavaScript

### Before Production Deploy

- [ ] Docker: CV service runs with `--cap-drop=ALL` and `--read-only` (except temp dir)
- [ ] Docker: `--tmpfs /app/uploads:rw,noexec,nosuid,size=2g`
- [ ] Docker: Memory limit `--memory=4g`
- [ ] nginx: `client_max_body_size 20m`
- [ ] CV service accepts connections only from Nuxt container (network isolation)
- [ ] SSL/TLS between Nuxt and CV service in production
- [ ] Model files stored outside web root, not directly accessible via HTTP
- [ ] Logging: all model uploads, deletions, and detection requests logged with userId

## Appendix A: API Contracts

### CV Service Endpoints

```
POST   /detect/image          # Upload photo, get detections
GET    /models                 # List all models
POST   /models                 # Upload new model (ONNX)
DELETE /models/{id}            # Delete model
GET    /health                 # Health check
```

### Nuxt API Routes

```
POST   /api/cv/detect/image           # Proxy to CV service + save session/detections
GET    /api/cv/models                  # List models (from DB)
POST   /api/cv/models                  # Upload model (admin, proxy to CV service + save to DB)
DELETE /api/cv/models/[id]             # Delete model (admin)
PUT    /api/cv/models/[id]             # Update model metadata (admin, set default)
GET    /api/cv/sessions                # List user's detection sessions
GET    /api/cv/sessions/[id]           # Get session with detections
DELETE /api/cv/sessions/[id]           # Delete session (own only)
GET    /api/cv/knowledge               # List knowledge base entries
POST   /api/cv/knowledge               # Create entry (admin)
PUT    /api/cv/knowledge/[id]          # Update entry (admin)
DELETE /api/cv/knowledge/[id]          # Delete entry (admin)
```
